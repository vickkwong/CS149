The main thing we did was make the outermost for loop parallel for each of the cpu_fft(x/y) and cpu_ifft(x/y). We had initially tried to make the inner y loops parallel but it not only did it not provide as much speedup but the output was also incorrect. This resulted from the fact that real_imag and imag_image values could not be altered until the first y loop finished running. We also tried to use "for schedule(dynamic)" and "for schedule(guided)" optimizations but we noticed that neither of those were as quick as "for" itself. We then combined the inner for loops where we iterate over n into one loop because there was no reason we couldn't do all the computations in one single loop. It would reduce the need to store fft_real and fft_imag arrays for each y loop. Then we noticed that cos(x) = cos(-x) and sin(x) = -sin(-x). Additionally the periods of each loop was n/size_x. Thus we were able to make all of these computations before hand and reuse the same values over and over again. We realized that a lot of computation time went into computing the cosine and sine values and thus witnessed a drastic drop in running time once we cut out those operations for every combination of n and y. We tried to make the threads share a realOutBuffer and imagOutBuffer but this was simply not possible since each thread presumably is working on a different part of the image. Finally the last optimization we made was computing the indices that neccessitated (x/y) * size_(x/y) only once under the assumption that multiplication was expensive but addition is cheap.

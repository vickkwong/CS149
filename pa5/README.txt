Our logic here is pretty simple. We basically took pa4's code and removed the outer two for loops. We realized that every element in the image matrix was doing the same exact computations so it would be possible to do those all simultaneously to speed up the program. We essentially created SIZE_X thread blocks with SIZE_Y threads in each block. Assuming that SIZE_X = SIZE_Y, it wouldn't really make a difference which constant we used for number of thread blocks and number of threads per block. Since we put into the kernels the method we want each thread to run, we simply removed the outer two for loops from pa4 and left in the one for loop where we actually calculate the new values for real_image and imag_image. We made the out-most for loop in each case iterate over the blocks and the second for loop in each iterate over the threads. This makes sense because each block is composed of threads. It is important that we synched the threads prior to reassigning values to real_image and imag_image because we would not want the values to change during our computation because for each iteration, the computations should be made with the old values of the image. Again we did the same things we did last week by precomputing all the values of sin and cosine (because they do repeat) and computing the values of indices only once where we can because multiplication is expensive where as addition is fast. To satisfy the shared requirement, we first pulled out the fft_real and fft_imag and made them shared across the threads in each thread block. When we noticed that this sped up the program a lot, we did the same thing with real_image and imag_image. Together, using these shared variables sped the program up a lot. Again this makes sense because each thread no longer has to maintain a local copy of it and that frees up space and time.
